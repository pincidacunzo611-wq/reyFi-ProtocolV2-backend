# Vault 服务 — 收益金库

## 1. 概述

Vault 服务管理 **自动化收益金库**。用户存入资产，金库通过预设的策略自动获得收益（如复利、套利、农场挖矿等）。

> **类比理解**：基金。你把钱交给基金经理（智能合约），他帮你投资赚钱。你持有的是基金份额（shares），基金的净值（NAV）会随着收益增长而增加。赎回时按净值换算回资产。

---

## 2. 核心概念

| 概念 | 解释 |
|------|------|
| **份额 Shares** | 存入资产后获得的凭证代币 |
| **NAV** | 净资产价值 = 总资产 / 总份额 |
| **APR/APY** | 年化收益率（单利/复利） |
| **策略 Strategy** | 金库的投资逻辑（如：存 USDC 到 Aave 赚利息） |
| **Harvest** | 执行策略收获利润（可由 Bot 定时触发） |
| **最大回撤** | 历史净值最大跌幅 |

---

## 3. 事件消费

| 事件 | 处理逻辑 |
|------|---------|
| **VaultCreated** | 写入 `vaults` 表 |
| **Deposit** | 更新 `vault_user_positions` 份额 + 记录活动 |
| **Withdraw** | 减少用户份额 + 记录活动 |
| **Harvest** | 写入 `vault_harvest_records`（收益金额、NAV变化） |
| **StrategyUpdated** | 更新策略配置 |

---

## 4. 数据库表

| 表 | 说明 |
|----|------|
| `vaults` | 金库基本信息（名称、资产、策略类型） |
| `vault_snapshots` | 定期快照（总资产、总份额、NAV、TVL、APR） |
| `vault_user_positions` | 用户持有的份额 + 成本基础 |
| `vault_harvest_records` | 每次 Harvest 的利润详情 |

# Bonds 服务 — 协议债券

## 1. 概述

Bonds 服务管理 **协议债券**。用户以折扣价购买代币，但需要经过一段 Vesting（归属期）才能全部解锁。

> **类比理解**：打折买股票。公司说"你现在花 90 元的 USDC 可以买到价值 100 元的代币，但 30 天内分批解锁"。这种机制帮助协议获得长期持有者，同时给买家一定折扣。

---

## 2. 核心概念

| 概念 | 解释 |
|------|------|
| **折扣率** | 债券价格比市场价低多少 (如 10%) |
| **Vesting 期** | 代币解锁的时间段 (如 30 天) |
| **已归属 Vested** | 已经解锁可领取的部分 |
| **可领取 Claimable** | 已归属但还没领取的部分 |
| **最大负债 Max Debt** | 一个债券市场最多能发行多少 |

---

## 3. 事件消费

| 事件 | 处理逻辑 |
|------|---------|
| **BondCreated** | 创建新的债券市场记录 |
| **BondPurchased** | 写入 `bond_positions` (status=vesting) + 活动流 |
| **BondRedeemed** | 写入 `bond_redemptions` + 更新仓位可领取金额 |

---

## 4. 数据库表

| 表 | 说明 |
|----|------|
| `bond_markets` | 债券市场（支付代币、产出代币、折扣率、归属期） |
| `bond_positions` | 用户购买记录（支付金额、产出金额、归属进度） |
| `bond_redemptions` | 赎回记录 |

# Governance 服务 — 治理

## 1. 概述

Governance 服务管理 **协议治理** 数据，实现去中心化决策。代币持有者可以创建提案、投票决定协议参数调整。

> **类比理解**：股东大会。持有 veToken（类似股份）的人可以提出议案（如：调整手续费比例），其他持有者投票赞成/反对/弃权。票数达到法定人数（Quorum）且多数赞成就通过执行。

---

## 2. 核心概念

| 概念 | 解释 |
|------|------|
| **veToken** | 投票锁仓代币，锁定时间越长投票权越大 |
| **提案 Proposal** | 一个治理议案（标题、描述、执行代码） |
| **投票** | 赞成(1) / 反对(0) / 弃权(2) |
| **法定人数 Quorum** | 提案通过需要的最低投票数 |
| **Gauge 投票** | 投票决定各池子的激励分配比例 |
| **Epoch** | 一个投票周期 (通常 1 周) |

---

## 3. 事件消费

| 事件 | 处理逻辑 |
|------|---------|
| **ProposalCreated** | 写入 `gov_proposals` (status=active) |
| **VoteCast** | 写入 `gov_votes` + 更新提案票数统计 |
| **ProposalExecuted** | 更新提案状态为 executed |
| **LockCreated** | 写入 `gov_ve_locks`（锁仓金额、解锁时间） |

---

## 4. 数据库表

| 表 | 说明 |
|----|------|
| `gov_proposals` | 治理提案（标题、状态、赞成/反对/弃权票数） |
| `gov_votes` | 投票记录（谁投了什么票、投票权重） |
| `gov_ve_locks` | veToken 锁仓记录（锁多少、解锁时间） |
| `gov_gauge_votes` | Gauge 投票（分配激励给哪个池子） |
| `gov_bribe_rewards` | 贿赂奖励记录 |
