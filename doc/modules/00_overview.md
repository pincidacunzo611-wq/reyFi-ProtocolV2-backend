# ReyFi Protocol V2 后端 — 整体架构概览

## 1. 项目定位

ReyFi Protocol V2 是一个 **全栈 DeFi 协议后端**，功能涵盖 DEX（去中心化交易所）、借贷、永续合约、期权、金库、债券、治理 7 大金融模块。

后端的核心职责是：
1. **监听区块链事件**：实时追踪链上合约的所有交易操作
2. **解析并存储数据**：将链上原始数据转为结构化业务数据
3. **对外提供 API**：为前端 DApp 提供高性能查询接口

> **类比理解**：后端就像一个"区块链搜索引擎"，它不断读取链上新区块，把重要的交易事件提取出来存到数据库里，让前端能快速查询，而不用每次都去链上读数据（链上读数据很慢且昂贵）。

## 2. 技术栈

| 技术 | 用途 | 为什么选它 |
|------|------|-----------|
| **Go** | 编程语言 | 高并发、编译快、适合后端微服务 |
| **go-zero** | 微服务框架 | 国产、自带服务发现/限流/熔断 |
| **MySQL** | 主数据库 | 存储所有业务数据，DECIMAL(65,18) 精度 |
| **Redis** | 缓存 | 加速热点数据查询 (如实时价格、交易对列表) |
| **Kafka** | 消息队列 | 服务间异步通信，解耦 Indexer 和业务服务 |
| **gRPC** | 内部通信 | 高性能二进制协议，服务间调用 |
| **Protobuf** | 接口定义 | 强类型、自动生成代码 |

## 3. 三层架构

```
┌─────────────────────────────────────────────────┐
│                   前端 DApp                       │
└──────────────────────┬──────────────────────────┘
                       │ HTTP/REST
┌──────────────────────▼──────────────────────────┐
│               Gateway (API 网关)                  │
│           端口 8888，统一入口                       │
│   JWT 鉴权 · 路由 · 限流 · CORS · 参数校验          │
└──────────────────────┬──────────────────────────┘
                       │ gRPC
┌──────────────────────▼──────────────────────────┐
│              RPC 服务层 (业务服务)                   │
│  ┌─────┐ ┌────────┐ ┌────────┐ ┌───────┐        │
│  │ DEX │ │Lending │ │Futures │ │Options│ ...     │
│  │9001 │ │ 9002   │ │ 9003   │ │ 9004  │        │
│  └──┬──┘ └───┬────┘ └───┬────┘ └───┬───┘        │
│     │ MySQL  │ MySQL    │ MySQL    │ MySQL      │
└─────┼────────┼──────────┼──────────┼────────────┘
      │        │          │          │
┌─────▼────────▼──────────▼──────────▼────────────┐
│                    Kafka                         │
│  reyfi.dex.events | reyfi.lending.events | ...   │
└──────────────────────┬──────────────────────────┘
                       │
┌──────────────────────▼──────────────────────────┐
│             Chain Indexer (链上索引器)              │
│         区块扫描 · 事件解析 · 重组检测               │
└──────────────────────┬──────────────────────────┘
                       │ ETH JSON-RPC
┌──────────────────────▼──────────────────────────┐
│                 区块链 (EVM)                       │
└─────────────────────────────────────────────────┘
```

**数据流向**（从下往上）：
1. **区块链** 产生新区块和交易
2. **Chain Indexer** 监听新区块，提取事件日志
3. **Kafka** 按模块分发事件消息
4. **各 RPC 服务** 消费事件，写入 MySQL
5. **Gateway** 接收前端请求，调用 RPC 服务查询数据

## 4. 11 个服务一览

| # | 服务 | 类型 | 职责 |
|---|------|------|------|
| 1 | **gateway** | HTTP API | 统一入口，路由到各 RPC |
| 2 | **chain-indexer** | 后台服务 | 监听链上区块，发布事件到 Kafka |
| 3 | **dex** | gRPC 服务 | 交易对、成交记录、K线、流动性 |
| 4 | **lending** | gRPC 服务 | 借贷市场、存借仓位、清算 |
| 5 | **futures** | gRPC 服务 | 永续合约市场、持仓、资金费率 |
| 6 | **options** | gRPC 服务 | 期权市场、持仓、波动率曲面 |
| 7 | **vault** | gRPC 服务 | 金库策略、份额管理、收益收获 |
| 8 | **bonds** | gRPC 服务 | 协议债券、Vesting 归属 |
| 9 | **governance** | gRPC 服务 | 治理提案、投票、veToken 锁仓 |
| 10 | **user** | gRPC 服务 | 钱包登录、资产汇总、活动流 |
| 11 | **bot** | 后台服务 | 清算监控、价格同步、日统计 |

## 5. 每个 RPC 服务的标准目录

```
app/<service>/
├── <service>.go                    # main() 入口
├── etc/<service>-dev.yaml          # 配置文件
├── rpc/<service>.proto             # gRPC 接口定义 (可选)
└── internal/                       # 私有业务代码
    ├── config/config.go            # 配置结构体
    ├── svc/servicecontext.go       # 依赖注入容器 (DB/Redis/Models)
    ├── model/model.go              # 数据模型 + SQL 操作
    ├── consumer/consumer.go        # Kafka 消费者 (处理链上事件)
    └── server/server.go            # gRPC 方法实现 (查询逻辑)
```

**核心设计原则**：
- `config` → 读配置
- `svc` → 初始化所有依赖（数据库、Redis、Model 实例）
- `model` → 纯数据库操作，不含业务逻辑
- `consumer` → 从 Kafka 读事件 → 调 Model 写数据库
- `server` → 从数据库读数据 → 返回给 Gateway

## 7. 如何本地运行

```bash
# 1. 启动基础设施
make dev-up   # MySQL + Redis + Kafka

# 2. 执行数据库迁移
make migrate

# 3. 启动服务（每个在一个终端）
make run-indexer    # 链上监听
make run-dex        # DEX gRPC
make run-user       # User gRPC
make run-gateway    # API 网关 (依赖上面的 RPC)
make run-bot        # 自动化任务
```

## 8. 关键概念速查

| 概念 | 解释 |
|------|------|
| **事件 (Event/Log)** | 合约执行时发出的日志，如 `Swap(sender, amount0, amount1, to)` |
| **Topic0** | 事件签名的 Keccak256 哈希，用于识别事件类型 |
| **幂等 (Idempotent)** | 同一个事件消费多次不会产生重复数据（通过 UNIQUE KEY 保证） |
| **链重组 (Reorg)** | 区块链分叉后回滚，我们的检测器会自动回滚受影响的数据 |
| **Health Factor** | 借贷仓位的健康度，< 1.0 会被清算 |
| **veToken** | 投票托管代币，锁仓时间越长投票权越大 |
| **NAV** | 金库净值 (Net Asset Value)，= 总资产 / 总份额 |
| **Funding Rate** | 永续合约的资金费率，多空双方按时间支付 |
