# 02 — 模块业务详解

> 本文档说明 ReyFi 后端每个模块负责什么、和哪些合约打交道、要同步哪些链上事件、要提供哪些能力给前端。

---

## 一、先建立整体认知

ReyFi 后端可以分成 3 类服务：

1. **链上索引层**
   负责监听合约事件，把链上变化翻译成数据库里的结构化数据。
2. **业务服务层**
   按模块拆分 RPC 服务，每个服务只关心自己的业务规则。
3. **对外接口层**
   由 API Gateway 暴露 REST 接口，给前端、运营后台、机器人使用。

从仓库现有合约来看，当前业务域主要包括：

- DEX
- Lending
- Futures / Leverage
- Options
- Vault
- Bonds
- Governance
- User / Account
- Bot / Automation

## 二、模块总表

| 模块 | 后端服务职责 | 对应核心合约 | 数据更新频率 |
|------|-------------|-------------|------------|
| Chain Indexer | 统一监听事件、做区块同步、重放、补块 | 所有协议合约 | 实时（每区块） |
| DEX | 交易对、池子、成交、流动性、路由报价 | `ReyFactory` `ReyPair` `ReyRouter` `LiquidityMining` | 高频（每笔交易） |
| Lending | 存款、借款、利率、健康度、清算记录 | `LendingPool` `RToken` `DebtToken` `LiquidationEngine` `CreditDelegation` | 中频 |
| Futures / Leverage | 永续仓位、保证金、资金费率、杠杆头寸 | `PerpetualMarket` `MarginManager` `FundingRate` `FuturesLiquidation` `LeverageEngine` `PositionManager` `VammLeverage` | 高频 |
| Options | 期权建仓、报价、结算、保护性策略 | `OptionsMarket` `OptionsPricing` `OptionsSettlement` `ProtectivePut` | 中频 |
| Vault | 金库产品、策略净值、申赎、收益分发 | `BaseVault` `VaultFactory` `StrategyManager` | 低频（按策略周期） |
| Bonds | 债券市场、折价发行、到期兑付、POL | `BondFactory` `BondNFT` `BondPricer` `ProtocolOwnedLiquidity` | 低频 |
| Governance | 提案、投票、锁仓、Gauge、Bribe | `Governor` `TimeLock` `GaugeController` `BribeMarket` `VeREY` | 低频（按治理周期） |
| User / Account | 钱包登录、资产总览、历史聚合、偏好设置 | `ReyAccountManager` `ReyRouter` `ReyPositionHelper` | 按需聚合 |
| Bot / Automation | 清算机器人、费率结算、策略再平衡 | 多模块合约协同 | 定时 + 事件驱动 |

## 三、Chain Indexer 模块

### 3.1 它的职责

Chain Indexer 是所有数据的入口，负责：

- 追踪最新区块高度
- 监听协议合约事件
- 做事件去重、重试、补块
- 检测链重组并回滚
- 把标准化事件投递到 Kafka

### 3.2 它不负责什么

- 不直接给前端提供查询接口
- 不承载复杂业务计算
- 不直接暴露协议级业务语义
- 不处理用户鉴权和授权

### 3.3 建议监听的事件类型

| 模块 | 重点事件 | Kafka Topic |
|------|---------|-------------|
| DEX | `PairCreated` `Mint` `Burn` `Swap` `Sync` `RewardPaid` | `reyfi.dex.events` |
| Lending | `Deposit` `Withdraw` `Borrow` `Repay` `Liquidate` | `reyfi.lending.events` |
| Futures | `PositionOpened` `PositionChanged` `PositionClosed` `FundingSettled` `Liquidated` | `reyfi.futures.events` |
| Options | `OptionPurchased` `OptionExercised` `OptionExpired` `SettlementExecuted` | `reyfi.options.events` |
| Vault | `VaultCreated` `Deposit` `Withdraw` `StrategyUpdated` `Harvest` | `reyfi.vault.events` |
| Bonds | `BondCreated` `BondPurchased` `BondClaimed` `BondRedeemed` | `reyfi.bonds.events` |
| Governance | `ProposalCreated` `VoteCast` `ProposalQueued` `ProposalExecuted` `GaugeVoted` | `reyfi.governance.events` |
| User | `Transfer` `LockCreated` `LockExtended` 等账户相关事件 | `reyfi.user.events` |

## 四、DEX 模块

### 4.1 业务目标

DEX 模块负责把链上 AMM 数据变成前端可直接消费的行情和交易数据。

### 4.2 主要能力

- 展示所有交易对和池子状态
- 查询池子储备、价格、24h 成交量
- 展示用户 LP 头寸和挖矿奖励
- 生成交换、加流动性、减流动性交易参数
- 输出 K 线、成交明细、TVL 排名

### 4.3 关键数据流

#### 读路径

```
前端请求 → Gateway → DEX RPC → Redis 缓存?
                                  │
                          ┌───────┴───────┐
                          ▼               ▼
                       命中缓存        未命中
                       直接返回     查 MySQL → 写缓存 → 返回
```

1. 前端查询 `/api/dex/pairs`
2. DEX 服务先读 Redis
3. 缓存未命中则读 MySQL 聚合表
4. 返回交易对列表和统计指标

#### 写路径

1. Indexer 捕获 `Swap` / `Mint` / `Burn`
2. 发送到 `reyfi.dex.events` 主题
3. DEX 消费者更新交易表、K 线表、池子快照表
4. 删除或刷新对应缓存

### 4.4 推荐子能力拆分

| 子域 | 说明 | 优先级 |
|------|------|-------|
| Pair Catalog | 交易对基础信息 | P0 |
| Market Data | K 线、成交、价格、TVL | P0 |
| Liquidity | LP 头寸、份额、收益 | P1 |
| Router Quote | 路径搜索、滑点计算、交易构造 | P0 |
| Mining | 流动性挖矿和奖励领取 | P2 |

### 4.5 关键计算逻辑

| 计算项 | 公式/方法 | 更新时机 |
|--------|----------|---------|
| 价格 | `reserve1 / reserve0` | 每次 Sync 事件 |
| 24h 成交量 | 滑动窗口聚合 `dex_trades` | 每次 Swap 事件 |
| TVL | `reserve0 * price0_usd + reserve1 * price1_usd` | 每次 Sync 事件 |
| LP 份额价值 | `用户LP / totalSupply * TVL` | 查询时实时计算 |
| 手续费 APR | `日手续费 / TVL * 365` | 每日聚合 |

## 五、Lending 模块

### 5.1 业务目标

把借贷池的链上状态整理成用户能理解的账户、利率、风险数据。

### 5.2 主要能力

- 资产池列表、利用率、借贷 APR
- 用户存款、借款、授信委托
- 健康因子、可借额度、清算价格
- 借款历史、还款历史、清算历史
- 构造存款、取款、借款、还款交易

### 5.3 重点计算

| 计算项 | 公式 | 说明 |
|--------|------|------|
| 利用率 | `总借款 / 总存款` | 决定利率曲线位置 |
| 借款利率 | 由利率模型合约定义（通常为分段线性） | 随利用率动态变化 |
| 存款利率 | `借款利率 * 利用率 * (1 - 储备系数)` | |
| 用户抵押价值 | `Σ (存款量 * 币种价格 * 抵押系数)` | |
| 用户借款价值 | `Σ (借款量 * 币种价格)` | |
| 健康因子 | `抵押价值 / 借款价值` | < 1 即可被清算 |
| 清算价格 | 反算使健康因子 = 1 时的标的价格 | 给用户做风险提示 |

### 5.4 关键事件

| 事件 | 处理逻辑 |
|------|---------|
| `Deposit` | 更新用户存款头寸、市场总存款 |
| `Withdraw` | 更新用户存款头寸、市场总存款 |
| `Borrow` | 更新用户借款头寸、市场总借款 |
| `Repay` | 更新用户借款头寸、市场总借款 |
| `CreditDelegated` | 记录授信关系 |
| `Liquidate` | 记录清算明细、更新双方头寸 |

## 六、Futures / Leverage 模块

### 6.1 为什么合并介绍

从仓库结构看，永续、杠杆和 vAMM 相关能力紧密耦合，后端可以先在一个模块内实现，再视吞吐量拆分。

### 6.2 主要能力

- 仓位列表和仓位详情
- 保证金变化和资金费率历史
- 开仓、加仓、减仓、平仓交易构造
- 爆仓预警和可清算列表
- 交易排行榜、持仓方向统计

### 6.3 重点实体

| 实体 | 说明 | 关键字段 |
|------|------|---------|
| Market | 每个永续市场的基础配置 | 标的、最大杠杆、维持保证金率 |
| Position | 用户仓位 | 方向、大小、开仓价、标记价、未实现盈亏 |
| Margin Account | 用户保证金账户 | 可用余额、冻结金额、维持保证金 |
| Funding Snapshot | 资金费率结算快照 | 费率、结算时间、累计费率指数 |
| Liquidation Record | 清算记录 | 被清算仓位、清算价格、罚金 |

### 6.4 后端特别要做的事

- 定时结算资金费率（通常每 8 小时）
- 实时计算未实现盈亏
- 给机器人产出可清算候选列表
- 把复杂合约状态折叠成用户友好的视图
- 维护标记价格和指数价格

### 6.5 关键计算

| 计算项 | 说明 |
|--------|------|
| 未实现盈亏 | `(标记价 - 开仓价) * 仓位大小 * 方向` |
| 保证金率 | `(保证金 + 未实现盈亏) / (标记价 * 仓位大小)` |
| 清算价格 | 使保证金率等于维持保证金率时的价格 |
| 资金费率 | 通常基于标记价与指数价的偏差 |

## 七、Options 模块

### 7.1 主要能力

- 期权产品列表
- 报价试算
- 买入、行权、结算交易构造
- 到期状态与收益分析
- Protective Put 组合策略展示

### 7.2 后端的核心价值

期权模块很多数据不是链上直接存好的，因此后端要补齐：

| 需要补齐的数据 | 来源/方法 |
|---------------|----------|
| 历史波动率 | 基于 DEX 价格数据计算 |
| 隐含波动率快照 | 反推链上报价得出 |
| 到期收益曲线 | 根据行权价和标的价格模拟 |
| 组合盈亏模拟 | 基于多个期权头寸组合计算 |
| 到期前风险提示 | 定时检查即将到期的期权 |
| 希腊字母 (Greeks) | Delta / Gamma / Theta / Vega |

### 7.3 关键事件

| 事件 | 处理逻辑 |
|------|---------|
| `OptionPurchased` | 创建用户期权头寸 |
| `OptionExercised` | 标记头寸为已行权，记录盈亏 |
| `OptionSettled` | 结算到期的期权池 |
| `OptionExpired` | 标记过期未行权头寸 |

## 八、Vault 模块

### 8.1 主要能力

- 金库列表、策略类型、TVL、净值
- 用户份额、申购赎回历史
- 策略收益、复投记录、回撤
- 生成申购和赎回交易

### 8.2 当前合约映射

| 合约 | 作用 | 后端需要关注的重点 |
|------|------|------------------|
| `BaseVault` | 金库主体 | 总资产、总份额、净值变化 |
| `VaultFactory` | 创建和注册金库 | 新金库上线事件 |
| `StrategyManager` | 管理策略 | 策略切换、权重调整 |
| `LendingStrategy` | 借贷收益策略 | Harvest 事件、APY 变化 |
| `LPCompoundStrategy` | LP 复投策略 | 复投频率、收益来源 |
| `LeveragedLPStrategy` | 杠杆 LP 策略 | 杠杆倍数、爆仓风险 |
| `VammDeltaNeutralStrategy` | Delta Neutral 策略 | 对冲状态、净敞口 |

### 8.3 后端要产出的核心指标

| 指标 | 计算方法 | 更新频率 |
|------|---------|---------|
| TVL | 金库总资产 * 底层资产价格 | 每小时 |
| 每份额净值 | 总资产 / 总份额 | 每小时 |
| 7d / 30d 年化收益 | `(当前净值 / N天前净值 - 1) * (365 / N)` | 每日 |
| 最大回撤 | 历史净值序列的最大回撤 | 每日 |
| 策略持仓分布 | 各子策略资金占比 | 每次再平衡 |

## 九、Bonds 模块

### 9.1 主要能力

- 展示债券市场和折价率
- 记录用户购买、归属、兑付
- 计算剩余可售额度和到期时间
- 支持协议自有流动性（POL）分析

### 9.2 关键实体

| 实体 | 关键字段 |
|------|---------|
| Bond Market | 市场地址、兑付资产、支付资产、折价率、归属期、剩余容量 |
| Bond Position / NFT | 持有者、购买金额、兑付金额、归属开始/结束时间 |
| Vesting Schedule | 已归属比例、可领取金额、下次归属时间点 |
| Redemption Record | 领取时间、领取金额、剩余未归属 |

### 9.3 后端重点

- 计算债券折价（对比市价和债券价格）
- 展示归属进度（线性归属或阶梯归属）
- 提醒即将可兑付的仓位
- 统计 POL 对协议流动性的贡献
- 计算债券 ROI（考虑折价和归属期）

## 十、Governance 模块

### 10.1 主要能力

- 钱包签名登录后查看投票权
- 提案列表、详情、状态流转
- 投票记录和投票分布
- VeREY 锁仓信息
- Gauge 权重投票和 Bribe 收益

### 10.2 关键流程

#### 提案生命周期

```
┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐    ┌─────────┐
│  创建    │ →  │  投票中  │ →  │  排队中  │ →  │  可执行  │ →  │  已执行  │
│ Created  │    │  Active  │    │  Queued  │    │ Ready   │    │ Executed│
└─────────┘    └────┬────┘    └─────────┘    └─────────┘    └─────────┘
                    │
                    ├─→ 未达标 → 已失败 (Defeated)
                    └─→ 提案人 → 已取消 (Canceled)
```

1. 用户创建提案
2. 提案进入投票期
3. 用户投票（赞成 / 反对 / 弃权）
4. 达到条件后排队到 Timelock
5. 到时间后执行

#### Gauge / Bribe 流程

```
用户锁仓 REY → 获得 veREY → 给池子投票 → 分配激励权重
                                  ↑
                            Bribe 市场提供额外奖励
```

1. 用户锁仓获得 veREY
2. 用户给池子投票分配权重
3. 协议按权重分发激励
4. Bribe 市场额外提供奖励

### 10.3 后端聚合价值

- 计算每个账户可用投票权（考虑衰减）
- 输出提案倒计时
- 汇总投票参与率
- 汇总 Gauge 权重分布
- 计算 Bribe 的有效年化回报

## 十一、User / Account 模块

### 11.1 为什么它必须独立存在

因为用户看到的不是某一个模块，而是"我在整个 ReyFi 里的总资产、总负债、总收益、总风险"。

### 11.2 主要能力

- 钱包签名登录与 JWT 下发
- 用户总资产总览
- 跨模块头寸聚合
- 历史交易流和通知
- 地址标签、偏好设置、收藏列表

### 11.3 它依赖哪些下游服务

| 下游服务 | 提供什么 | 调用场景 |
|---------|---------|---------|
| DEX | LP 和交易历史 | 资产总览、交易流 |
| Lending | 存借款和健康度 | 资产总览、风险提示 |
| Futures | 仓位和 PnL | 资产总览、盈亏统计 |
| Options | 期权仓位 | 资产总览 |
| Vault | 份额和净值 | 资产总览 |
| Bonds | 债券资产 | 资产总览 |
| Governance | veREY 和投票权 | 治理信息 |

### 11.4 用户资产聚合策略

```
请求 "我的总资产"
       │
       ├──→ 并行调用 7 个 RPC 服务
       │         │
       │    ┌────┼────┬────┬────┬────┬────┐
       │    ▼    ▼    ▼    ▼    ▼    ▼    ▼
       │   DEX  Lend Futr  Opt Vault Bond  Gov
       │    │    │    │    │    │    │    │
       │    └────┴────┴────┴────┴────┴────┘
       │              │
       │         汇总结果
       │              │
       ▼              ▼
  缓存 10~30 秒    返回给前端
```

> **性能提示：** 使用 `goroutine + WaitGroup` 并行调用，设置单个 RPC 超时 3 秒，总超时 5 秒。任一服务超时不阻塞其他服务，降级返回部分数据。

## 十二、Bot / Automation 模块

### 12.1 主要职责

| 任务 | 触发方式 | 频率 |
|------|---------|------|
| 检查可清算账户 | 定时 + 价格变动事件 | 每 10 秒 |
| 执行清算 | 发现可清算时立即执行 | 事件驱动 |
| 资金费率结算 | 定时 | 每 8 小时 |
| Vault 再平衡和 Harvest | 定时 | 每 4~24 小时 |
| 扫描失败任务并补偿 | 定时 | 每 5 分钟 |
| 生成运维告警 | 异常检测 | 实时 |

### 12.2 为什么单独做服务

- 需要常驻运行
- 和用户请求解耦
- 需要任务调度、重试、幂等
- 可能需要独立密钥和风控策略
- 需要独立监控和告警

### 12.3 Bot 安全设计

| 安全措施 | 说明 |
|---------|------|
| 独立热钱包 | 不和其他服务共用密钥 |
| 单笔限额 | 每笔链上操作设置最大金额 |
| 日限额 | 每日 Gas 消耗上限 |
| 白名单 | 只允许调用指定合约 |
| 审计日志 | 所有链上操作可追溯 |

## 十三、推荐开发优先级

### Phase 1 — 最小可用（先让"看得到 + 能交易"跑通）

- Chain Indexer
- DEX
- User

先把"可查询、可展示、可交易构造"跑通。

**验收标准：** 前端能展示交易对列表、成交记录，用户能通过 API 构造 Swap 交易。

### Phase 2 — 接入风控（引入有风险的模块）

- Lending
- Futures / Leverage

开始接入风险和仓位逻辑。

**验收标准：** 用户能查看借贷头寸和健康度，能查看期货仓位和盈亏。

### Phase 3 — 补齐生态（完整 DeFi 矩阵）

- Vault
- Bonds
- Governance
- Bot

补齐收益类、治理类、自动化能力。

**验收标准：** 所有模块数据可查，Bot 能自动执行清算和结算。

## 十四、一句话记忆

- **Indexer** 负责"听见链上发生了什么"
- **各业务服务** 负责"理解这些变化意味着什么"
- **User 服务** 负责"把所有模块拼成用户视角"
- **Bot 服务** 负责"自动执行那些不能等用户手动点按钮的事"
- **Gateway** 负责"把所有能力安全、高效地暴露给外部"
